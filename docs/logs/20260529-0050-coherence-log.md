## Original request

Fix the cross-process cache-coherence gap between cq's server and the
`cq-mcp` stdio binary. Both processes open independent `FsLedgerStore`
instances against the same `docs/` directory. On-disk consistency is
maintained by the existing advisory-lockfile mechanism (D-LED), but
**in-memory caches are not invalidated when the other process writes**.
A Codex session that creates a ledger via `mcp__cq__create_ledger` is
invisible to cq server until restart.

Ship a per-process internal WebSocket channel that delivers
`ledger.changed` invalidation notifications between the two processes.

## Surprises / constraints future work must respect

1. **WsData discriminated union.** The browser-facing `WsSessionData`
   has no `kind` field; the internal `InternalWsConnData` does. We
   branch at runtime via `"kind" in ws.data && ws.data.kind ===
   "internal"`. Future browser-data shape changes must NOT introduce a
   `kind` field that collides with `"internal"`.

2. **Bridge.setInternalWsUrl is one-shot, called pre-first-Codex.**
   The Bridge constructs the CodexBridge lazily on first Codex
   ChatStart, so `setInternalWsUrl` must be called BEFORE any client
   can reach the WS handler. `server.ts` and `devServer.ts` do this
   between `Bun.serve(...)` returning and any external trigger; the
   invariant holds because `Bun.serve` blocks `accept(2)` until the
   socket is bound.

3. **No reconnection logic.** cq-mcp is short-lived per Codex session.
   On disconnect we log and continue serving stdio MCP; cache
   invalidation is silently disabled for the rest of the session.
   If we later support long-lived Codex sessions, add a backoff +
   reconnect loop in `InternalWsChannel`.

4. **The integration test spawns cq-mcp via `bun run` on the source.**
   The Nix-built bin works in standalone mode (smoke-tested) but the
   integration test uses the dev path so it doesn't depend on
   `nix build`-having-run. The two paths share the same code; if they
   diverge, add a Nix-bin variant of the integration test.

5. **Token leaks in env.** The token is passed via `CQ_INTERNAL_WS_TOKEN`
   to cq-mcp. Anyone with `ps eww` access on the same machine can read
   it. The threat model is single-user dev/VPN deployment per the
   brief; this is acceptable. For multi-tenant deployments,
   socket-pair-with-FD-handoff would be a better path.
