# Plan-flow: portable goal→reviewed-plan workflow (milestone + PR plan)

Status: draft (planning). Author: opus-4.8[1m]. Date: 2026-06-01.

Drives a user's goal from a one-line statement to a reviewed, fine-grained
implementation plan, stored entirely in the existing ledgers, working
identically under Claude Code and Codex. Formality lives in the MCP server +
ledger data, never in an orchestrator prompt. Lifecycle keyed to the `goals`
ledger phases: `clarifying → planning → planned → building → done/abandoned`.

This doc grounds every PR in real file paths / function names found in the repo.

## Cross-cutting architectural notes (LOCKED)

These hold for the whole programme. Each names the PR where it lands.

- **(A1) F1 belongs in `core.ts::applyUpdateItem`, not the stores.** Both
  `FsLedgerStore.updateItem` (`packages/ledger/src/store/FsLedgerStore.ts:441`)
  and `InMemoryLedgerStore.updateItem`
  (`packages/ledger/src/store/InMemoryLedgerStore.ts:287`) delegate the actual
  status mutation to the pure helper `applyUpdateItem`
  (`packages/ledger/src/store/core.ts:209`). Injecting the transition guard
  there gives both stores identical behaviour for free and keeps the dual-store
  parity invariant true by construction. Lands in **PR1**.

- **(A2) Validate the transition only when status changes.** `applyUpdateItem`
  already branches on `patch.status !== undefined` (core.ts:216). Field-only
  updates (the common case: writing an `answer`, a `criticism`, etc.) must skip
  the guard. Self-transitions (`x→x`) are allowed (no-op). Lands in **PR1**.

- **(A3) `transitions` is an OPTIONAL schema field; absent ⇒ no transition
  enforcement (back-compat).** Shape: `transitions?: Record<string,string[]>`
  on `LedgerSchema` (`packages/ledger/src/types.ts:22`). Key = from-status,
  value = allowed to-statuses. When `transitions` is undefined the guard is a
  no-op so every non-canonical / user-created ledger keeps working unchanged.
  Lands in **PR1**.

- **(A4) `transitions` must round-trip through `ledgers.yaml` AND survive the
  divergence guard.** The schema is persisted ONLY in the registry
  (`docs/ledgers.yaml`), not in per-ledger `.md` frontmatter — `parseLedger`
  takes the schema as a parameter, so the markdown parser needs NO change.
  Four touch-points must learn about `transitions` together or it silently
  drops / falsely trips the guard:
  - `serializeRegistry` (`packages/ledger/src/registry.ts:142`) — emit it.
  - `parseSchema` (`packages/ledger/src/registry.ts:78`) — read + validate it.
  - `schemasEqual` (`FsLedgerStore.ts:951`) — compare it (divergence guard).
  - `cloneSchema` (`InMemoryLedgerStore.ts:567`) — copy it (else fetched views
    lose it).
  Plus `validateSchema` (`core.ts:113`) and the two Zod `schemaSchema`
  fragments (`ledgerTools.ts:104`, `stdioLedgerTools.ts:73`) for `create_ledger`.
  Lands in **PR1**.

- **(A5) F2 preconditions belong at the LedgerStore layer (in BOTH stores),
  not the pure core and not only the MCP tool layer.** Rationale: the existing
  precedent is `archiveMilestone`/`assertMilestoneActive`
  (`core.ts:538`, called from `FsLedgerStore.createItem:467` and
  `InMemoryLedgerStore.createItem:308`) — cross-ledger invariants are enforced
  where the cross-ledger VIEW exists, i.e. the store. The pure `core.ts`
  helpers operate on a single `Ledger` and cannot see the questions/decisions
  ledgers. Putting it in the MCP tool layer would leave the TUI/web editor (a
  direct MCP client) and any future caller able to bypass it. So: add a private
  `assertGoalsTransitionAllowed(...)` to each store's `updateItem`, gated on
  `ledgerId === GOALS_LEDGER && patch.status !== undefined`, that reads the
  goals item + the questions/decisions ledgers from the in-memory maps. Lands
  in **PR2**.

- **(A6) F3: add a NEW canonical `reviews` ledger (idPrefix `R`), not fields on
  an existing kind.** Rationale grounded in this repo: (1) schema validation
  cannot express "this string field must be one of go-ahead|revise" — field
  types are only `string|string[]|id|id[]|timestamp` (`types.ts:13`) — BUT
  `statusValues` IS an enum the store enforces (`assertStatusAllowed`,
  `core.ts:565`). Modelling the verdict as the review item's `status`
  (`statusValues: ["go-ahead","revise"]`) gives schema-enforced verdicts for
  free, exactly the F3 requirement, with no new validation machinery. (2) A
  dedicated ledger keeps the adversarial-review history queryable
  (`fts_search ledger:reviews`) and linkable to the goal via `ledgerRefs`,
  mirroring how `questions`/`decisions` already work. (3) Reusing `decisions`
  or `hypothesis` would overload their status state-machines and pollute their
  semantics. `new_questions`/`criticism` are `string[]` fields; link to the
  goal via the shared `ledgerRefs: id[]`. Lands in **PR3** (part of milestone 2;
  listed here because the decision is cross-cutting).

- **(A7) Author-enum: NO hard enforcement; keep `author` free-form.** `author`
  is intrinsic, optional, free-form (`types.ts:66`, author-session.test.ts) and
  deliberately NOT a schema field so it never trips the divergence guard. A hard
  allowed-author enum would have to live in the store and would couple the
  library to a fixed model roster — wrong layer, high churn. Instead document
  the convention in the prompts (already in `SERVER_INSTRUCTIONS`,
  `main.ts:171`) and, if drift becomes a problem, add a SOFT lint later. The
  model derives its own class string at runtime; never hardcode it in a prompt.
  Decision only; no code. Recorded here.

- **(A8) Command/agent files do not exist yet and must be created.** There is
  NO `.claude/commands/`, NO `.claude/agents/`, NO `.codex/`, NO `AGENTS.md`.
  Only `.claude/settings.local.json` + `.claude/worktrees/` exist. The repo is
  wired to its own ledger via `.mcp.json` (server name `ledger` ⇒ tools are
  `mcp__ledger__*`). Proposed paths (PR in milestone 2):
  - `.claude/commands/plan/start.md`, `.claude/commands/plan/advance.md`
    (Claude slash-commands `/plan:start`, `/plan:advance`).
  - `.claude/agents/plan-advance.md`, `.claude/agents/plan-reviewer.md`
    (subagent definitions with the tool policies from the brief).
  - Codex mirror: `.codex/prompts/plan-start.md`,
    `.codex/prompts/plan-advance.md`, `.codex/prompts/plan-reviewer.md`, and a
    top-level `AGENTS.md` documenting the portable workflow + provenance rule.
    (Confirm exact Codex prompt-dir convention with the USER — see open Q.)

- **(A9) F4 watcher reuses `@cq/ledger-live` `LiveManager` + the HTTP `/mcp`
  read endpoint.** It does NOT read files. `LiveManager`
  (`packages/ledger-live/src/index.ts:84`) already gives a resilient `/ws`
  client whose `onChanged(ledger)` fires on the `{type:"changed",ledger}` frame
  the HTTP server publishes (`main.ts:290` `changedFrame`, `WS_PATH=/ws`). The
  watcher reads ledger state via the MCP Streamable-HTTP endpoint
  (`MCP_HTTP_PATH=/mcp`, `main.ts:47`) calling `fetch_ledger goals` +
  `fetch_ledger questions`. It evaluates the resume predicate and, on a trip,
  re-invokes the open session / prints the goalId. It MUST `check()` once
  immediately on arm (before subscribing settles) to close the race where
  answers land while disarmed. Lands in **PR(milestone 3)**.

## Milestone breakdown (high level)

- **M1 — Server/data formality (F1+F2+F3).** The load-bearing, fully
  test-covered core: transition guard, cross-ledger preconditions, the
  `reviews` ledger. Pure `@cq/ledger` work; no prompts, no new process. This is
  the milestone broken down to PR level below. Everything else depends on it.

- **M2 — Portable orchestration prompts.** `/plan:start`, `/plan:advance`
  (thin main-session loop spawning `plan-advance` + `plan-reviewer`), the two
  subagent definitions, and the Codex mirror + `AGENTS.md`. Pure
  prompt/markdown files; depends on M1's tool/data semantics being final.

- **M3 — Non-AI watcher/driver (F4).** The Bun/TS process subscribing to
  ledger-live and re-invoking the open session on the resume predicate. Depends
  on M1 (predicate reads the F1/F2-governed goal/questions state).

Sequencing: M1 → M2 → M3. M2 and M3 can overlap once M1 lands.

## M1 — PR-level breakdown (dependency order)

Repo convention (CLAUDE.md): "Reproduce a defect (failing test or documented
repro) before fixing it." Each PR therefore lands its failing test FIRST in the
same PR, before the implementation, and ends green. Run from repo root.

### PR1 — F1 transition guard + `transitions` schema plumbing

Scope: introduce the optional `transitions` map on `LedgerSchema`, plumb it
through every schema touch-point (A4), and enforce it in `applyUpdateItem`
(A1/A2/A3). Add `transitions` maps to the canonical schemas that have a real
phase machine — at minimum `goals`
(`clarifying→[planning,abandoned]`, `planning→[planned,clarifying,abandoned]`,
`planned→[building,abandoned]`, `building→[done,abandoned]`); decide per-ledger
for tasks/defects/hypothesis/questions/decisions/milestones (propose: encode
the natural forward machine for each; `terminalStatuses` already prevents
leaving terminal so transitions out of terminal default to `[]`).

Failing-test-first:
- New `packages/ledger/test/transitions.test.ts`: a dual-store suite (mirror
  the `inMem`/`fs_` factory pattern from `canonical-ledgers.test.ts`) asserting
  (a) a legal goals transition `clarifying→planning` succeeds; (b) an illegal
  `clarifying→planned` throws a new `IllegalTransitionError`; (c) a field-only
  update on a `clarifying` goal succeeds (guard skipped, A2); (d) a self
  `x→x` is allowed; (e) a ledger whose schema has no `transitions` allows any
  in-`statusValues` change (back-compat, A3).
- Extend `packages/ledger/test/canonical-ledgers.test.ts` divergence test
  expectation if needed (it already asserts a superset trips the guard; adding
  `transitions` must keep that test green — verify `schemasEqual` change does
  not break it).

Target files:
- `packages/ledger/src/types.ts` (add `transitions?` to `LedgerSchema`; add
  `IllegalTransitionError extends LedgerError`).
- `packages/ledger/src/store/core.ts` (`applyUpdateItem` guard;
  `validateSchema` — validate that `transitions` keys/values ⊆ `statusValues`).
- `packages/ledger/src/constants.ts` (add `transitions` to `GOALS_SCHEMA` and
  the other canonical schemas).
- `packages/ledger/src/registry.ts` (`parseSchema` read+validate;
  `serializeRegistry` emit).
- `packages/ledger/src/store/FsLedgerStore.ts` (`schemasEqual`).
- `packages/ledger/src/store/InMemoryLedgerStore.ts` (`cloneSchema`).
- `packages/ledger/src/mcp/ledgerTools.ts` + `mcp/stdioLedgerTools.ts`
  (`schemaSchema` Zod gains optional `transitions`).
- `packages/ledger/src/index.ts` (export `IllegalTransitionError` if it is a
  named export consumers need; it's in `types.ts` which is `export *`).

Acceptance criteria:
- `bun test packages/ledger/test/transitions.test.ts` → all new cases pass
  against BOTH stores.
- `bun test packages/ledger/test/canonical-ledgers.test.ts` → still green
  (round-trip + divergence guard unaffected by the new field).
- `bun run check` (tsc -b + eslint + bun test) → clean.

NOTE (migration risk): adding `transitions` to canonical schemas changes their
canonical shape, so the already-bootstrapped dogfood `docs/ledgers.yaml`
(present, 6.1 KB) will DIFFER from the new canonical schema and the divergence
guard (`FsLedgerStore.init`, FsLedgerStore.ts:279) will refuse to start. See
Risks — needs a one-time registry refresh or a divergence-guard allowance for
"on-disk is a strict subset of canonical (missing only transitions)".

### PR2 — F2 server-enforced goal preconditions

Scope: enforce, in BOTH stores' `updateItem` (A5), the two locked preconditions
when a `goals` item changes status:
- (a) a goal cannot LEAVE `clarifying` while any `open` `questions` item links
  to it via `fields.ledgerRefs` containing `goals:<G>`;
- (b) a goal cannot ENTER `planned` without a linked `decisions` item in a
  locked/approved status (`status:"locked"`) referencing it
  (`ledgerRefs` contains `goals:<G>`, OR the goal's own `decisions`/`ledgerRefs`
  link — pick the link direction during impl; recommend questions/decisions →
  goal via `ledgerRefs`, matching the brief's batch-filing direction).

Failing-test-first:
- New `packages/ledger/test/goal-preconditions.test.ts` (dual-store): build a
  goal in `clarifying`, file an `open` question with
  `fields:{ledgerRefs:["goals:G1"]}`, assert `updateItem(goals,G1,{status:"planning"})`
  throws a new `GoalPreconditionError`; answer the question
  (`status:"answered"`) and assert the transition now succeeds. Then assert
  `clarifying`→`planning`→attempt `planned` throws until a `locked` `decisions`
  item linking `goals:G1` exists, after which it succeeds.

Target files:
- `packages/ledger/src/types.ts` (`GoalPreconditionError`).
- `packages/ledger/src/store/FsLedgerStore.ts` (private precondition check in
  `updateItem`, reading `this.ledgers.get(QUESTIONS_LEDGER|DECISIONS_LEDGER)`).
- `packages/ledger/src/store/InMemoryLedgerStore.ts` (same check, same source
  of truth — `this.ledgers`).
- Optionally a shared pure helper in `core.ts` taking the three ledgers
  explicitly (keeps the two store copies identical — preferred, mirrors how
  `assertMilestoneActive` is a core helper the stores call).
- `packages/ledger/src/constants.ts` (import GOALS/QUESTIONS/DECISIONS names —
  already exported).

Locking note: the goals/questions/decisions reads happen under the goals
per-ledger lock; since these are cross-ledger reads, follow the documented
lock order (`__milestones__` first, then per-ledger alphabetical) ONLY if a
write to another ledger is taken — here it's read-only of in-memory state under
the goals lock, which is safe (no other lock needed, reads are synchronous
against the in-memory map). Confirm against the concurrency test patterns in
`packages/ledger/test/concurrency.test.ts` during impl.

Acceptance criteria:
- `bun test packages/ledger/test/goal-preconditions.test.ts` → passes on BOTH
  stores.
- `bun run check` → clean.

### PR3 — F3 `reviews` canonical ledger + schema-validated verdict

Scope: add a canonical `reviews` ledger (A6). Schema:
`statusValues:["pending","go-ahead","revise"]`, `terminalStatuses` per design
(propose `["go-ahead","revise"]` so a finished review is terminal/archivable),
`idPrefix:"R"`, fields `{ new_questions: string[], criticism: string[],
...COMMON_REF_FIELDS }` (link to goal via `ledgerRefs`). Add it to
`CANONICAL_LEDGERS` (constants.ts:193). The verdict is the item STATUS (enum
enforced by `assertStatusAllowed`), satisfying F3 without new validation code.
Optionally give it a `transitions` map (`pending→[go-ahead,revise]`).

Failing-test-first:
- Extend `packages/ledger/test/canonical-ledgers.test.ts` CASES with a
  `reviews` case (create with `new_questions`/`criticism` arrays, update status
  to `go-ahead`, fetch/search/archive) — this reuses the existing dual-store
  lifecycle harness and the divergence-guard + parser-roundtrip loops, so one
  added CASE covers create/update/fetch/search/archive AND on-disk round-trip.
- Assert an out-of-enum verdict (`updateItem(reviews,R1,{status:"approved"})`)
  throws `InvalidStatusError`.

Target files:
- `packages/ledger/src/constants.ts` (`REVIEWS_LEDGER`, `REVIEWS_SCHEMA`, add
  to `CANONICAL_LEDGERS`).
- `packages/ledger/src/index.ts` (export the new name + schema, matching the
  existing canonical exports).
- `packages/ledger/test/canonical-ledgers.test.ts` (new CASE + count updates:
  the `enumerate()` and id-prefix assertions in store-abstract.ts /
  canonical-ledgers.test.ts count ledgers — update `["D","G","H","K","Q","T"]`
  to include `"R"`).
- `packages/ledger/test/store-abstract.ts` (the `BOOTSTRAPPED` list derives from
  `CANONICAL_LEDGERS` so it auto-updates; verify the `enumerate()` test still
  passes — it should, as it maps from the constant).

Acceptance criteria:
- `bun test packages/ledger/test/canonical-ledgers.test.ts` → reviews CASE
  passes on both stores; enum-rejection assertion passes.
- `bun test packages/ledger/test/store-fs.test.ts packages/ledger/test/store-inmemory.test.ts`
  → `enumerate()` test green with `reviews` included.
- `bun run check` → clean.

### PR4 — (M1 closeout) divergence-guard migration for canonical schema bump

Scope: because PR1+PR3 change canonical schemas, provide the one-time path so a
pre-existing `docs/ledgers.yaml` does not brick server start. Recommended:
treat an on-disk schema that is identical except for ADDED `transitions` and/or
is a strict subset of the canonical schema as "needs upgrade, rewrite in place"
rather than "divergent, refuse". Implement in `FsLedgerStore.init`
(FsLedgerStore.ts:274-287) by replacing the binary `schemasEqual` gate with:
equal ⇒ ok; on-disk is a forward-compatible subset of canonical ⇒ adopt
canonical + mark registry dirty (rewrite); otherwise ⇒ refuse (preserve the
existing loud-divergence test). USER decision may simplify this to "just
re-bootstrap the dogfood ledger by hand" — see open Q.

Failing-test-first:
- New test in `canonical-ledgers.test.ts`: write a registry with the OLD
  (transitions-less) canonical schemas, init, assert it starts AND the rewritten
  `ledgers.yaml` now carries `transitions`.
- Keep the existing "divergence guard fires for a hand-edited schema" test green
  (a genuinely incompatible schema — e.g. extra status with no upgrade path —
  must still throw).

Target files:
- `packages/ledger/src/store/FsLedgerStore.ts` (init upgrade logic;
  `schemasEqual` → a `schemaUpgradeDecision()` helper).
- `packages/ledger/test/canonical-ledgers.test.ts`.

Acceptance criteria:
- `bun test packages/ledger/test/canonical-ledgers.test.ts` → upgrade test
  passes; loud-divergence test still passes.
- Manual: starting `ledger-mcp` against the existing repo `docs/` no longer
  throws `different schema` (verify by running the stdio server briefly).
- `bun run check` → clean.

## Risks / assumptions / open questions

- **[!] BLOCKER — canonical schema bump bricks existing dogfood ledger.** The
  repo's own `docs/ledgers.yaml` is already bootstrapped; PR1/PR3 change
  canonical schemas, so `FsLedgerStore.init`'s divergence guard will refuse to
  start until PR4's upgrade path lands OR the file is regenerated. USER: do you
  want (a) an automatic forward-compatible upgrade in `init` (PR4 as scoped), or
  (b) a manual one-time regeneration of `docs/ledgers.yaml`? This changes PR4's
  scope.
- **[!] Codex prompt/agent file convention.** Need the USER to confirm where
  Codex looks for portable prompts/subagents (`.codex/prompts/`? `AGENTS.md`
  only? a different dir). A8 assumes `.codex/prompts/` + `AGENTS.md`.
- **Assumption — `reviews.terminalStatuses`.** Proposing `["go-ahead","revise"]`
  terminal so a completed review archives with its milestone; if reviews should
  stay mutable, make only `go-ahead` terminal. Low-risk, decide in PR3.
- **Assumption — transition maps for non-goal ledgers.** PR1 proposes encoding
  the natural forward machine for tasks/defects/etc.; if the team prefers to
  scope F1 to `goals` ONLY for now, the other canonical schemas keep
  `transitions` undefined (A3 makes that a no-op) and the schema bump shrinks to
  just `goals` + `reviews`, reducing the PR4 migration surface.
- **Link direction (F2b).** Recommend questions/decisions → goal via
  `ledgerRefs:["goals:<G>"]` (matches the brief's "file questions linked to the
  goal"); confirm in PR2.
- **Lock discipline for F2 cross-ledger reads.** The preconditions read other
  ledgers' in-memory state under the goals lock; verified read-only so no extra
  lock is required, but cross-check `concurrency.test.ts` invariants in PR2.
