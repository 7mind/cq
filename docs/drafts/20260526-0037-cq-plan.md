## 0. Goal & non-negotiables (recap)

**Goal.** Build cq, a TypeScript web application that lets the user
drive the Claude Agent SDK from a browser tab over WebSocket,
replicating the Claude CLI experience faithfully, with a second tab
that surfaces the full history of sub-agent invocations in
human-readable form `[brief § 1]`.

**Hard constraints carried forward verbatim from `[brief § 3]`:**

- TypeScript strict mode throughout. No `any` except at external
  library boundaries with a justifying comment.
- Bun runtime (`bun:sqlite`, `Bun.serve` WebSocket handler, `bun test`,
  Bun bundler). Node-shaped APIs only where Bun lacks coverage.
- React + TypeScript, functional components + hooks, no class
  components.
- `@anthropic-ai/claude-agent-sdk` latest stable on the **server only**;
  browser never imports it.
- WebSocket-only transport for application data. Static assets HTTP-served
  by the same Bun process.
- No auth, no CORS-permissive flows, no login. Binds to VPN interface
  via CLI flag.
- Single active session at a time; pool size = 1. New chat replaces
  current (with confirm if mid-stream).
- Working directory fixed at startup via `--cwd <path>`. UI displays
  read-only. No directory picker.
- MCP servers from `~/.claude/` config forwarded to each query.
- Skills from `~/.claude/skills/` surfaced via slash commands.
- Plugins and hooks out of scope for v1.
- Two tabs only: Chat and History.
- No telemetry, no analytics, no remote logging. Local logs only.

**Tightenings proposed by S4 (none weaken a non-negotiable):**

- **Bun pinned to 1.3.13** (the version the flake provides today; bump
  is a deliberate cycle, not a drift).
- **React 19.2.x** (current stable; React 19 is the only major that
  ships native hooks for `use()` and improved Suspense, which we will
  use for the History detail-view lazy loader). The brief allows
  "React + TypeScript" with no major version constraint; React 19 is
  unambiguously the recent stable per `[ccmd §1] Recent stable versions`.
- **Styling: CSS modules.** Justification: zero runtime, native Bun
  bundler support, smallest variety budget vs Tailwind (which would
  add a build step) and vanilla-extract (which would add a compile
  step). Two-sentence justify lives in § 1.
- **Zod pinned to 4.x** because the SDK declares `zod ^4.0.0` as a
  peer dep `[sdk:package.json]`. Zod 3 will break SDK type
  compatibility.
- **Single CLI binary, no separate dev/prod variants.** `bun run start`
  and `bun run dev` differ only in dev-mode toggling; both serve from
  one `Bun.serve`. (G2c: F-12) `dev` configures `Bun.serve({
  development: { hmr: true } })` (Bun 1.3 integrated HMR); `start`
  serves the pre-built bundle. We do NOT use `bun build --watch`
  (that's rebuild-and-full-reload, not HMR).

**S4-imposed scope freezes (will be defended at G2b review):**

- No directory picker, ever. (`[brief § 3]`).
- No "list of sessions stored on disk" feature beyond the History tab.
  History rows ARE the sessions list. There is no separate "Sessions"
  view.
- No keybinding configurability in v1 beyond the three required
  (Cmd/Ctrl+Enter, Shift+Enter, Esc) `[brief § 4]`.
- No themes. One dark theme — the brief does not ask for two.

## 3. Wire protocol

Every frame is a JSON object validated by a discriminated-union Zod
schema. Top-level discriminator: `type` (string). All schemas live
in `packages/shared/src/protocol.ts` `[brief § 7]`.

### 3.1 Common envelope fields

Every frame carries:
- `type: string` — discriminator.
- `seq: number` — monotonic per-session sequence number; client and
  server each maintain their own outbound counter. The other side
  echoes the last-received counter in `ack` on heartbeats so a
  reconnect can detect gaps `[brief § 6]`.
- `ts: number` — `Date.now()` at send time; used for diagnostics, not
  authority.

### 3.2 Heartbeat (resilient-ws-ui R3, R11, V4)

```ts
// Both directions; matches R3 nonce-correlated pattern.
ClientHbPing = z.object({
  type: z.literal("hb.ping"),
  seq: z.number().int().nonnegative(),
  ts: z.number(),
  nonce: z.string().length(16),     // 8 random bytes hex-encoded
  ackSeq: z.number().int().nonnegative().nullable(),
});
ServerHbPong = z.object({
  type: z.literal("hb.pong"),
  seq: z.number().int().nonnegative(),
  ts: z.number(),
  echoNonce: z.string().length(16),
  clientTs: z.number(),
  serverTs: z.number(),
});
// And the symmetric server-initiated ping for R11.
ServerHbPing = z.object({ type: z.literal("hb.sping"), seq, ts, nonce });
ClientHbPong = z.object({ type: z.literal("hb.spong"), seq, ts, echoNonce, serverTs });
```

The server-initiated ping uses a different `type` tag so client and
server don't accidentally route a pong to the wrong matcher. The
**server keeps `currentNonce` and `previousNonce` per connection** and
accepts an `hb.spong` that echoes either — this is the one-tick
lookback from `[ws R11]`.

`SDKMessageEnvelope` is a zod-validated **passthrough** of the SDK's
own `SDKMessage` discriminated union `[sdk:sdk.d.ts:SDKMessage]`. We
do NOT re-shape the SDK types — we forward them tagged by their
`type`/`subtype` so the renderer sees exactly what the SDK emits.
The shared package defines the Zod shape just well enough to
distinguish each variant by `{type, subtype?}`; unknown variants
fail-open at the renderer (display as a raw JSON card) but pass
schema validation as `passthrough()` records. This is the
fail-fast-at-boundary / surface-failure-in-UI tension from
`[ccmd §1]` resolved deliberately: schema validation rejects
malformed frames, but tag-unknown valid frames render as "unknown
SDK event vN" rather than crash. New SDK message types are surfaced
visibly in the UI as a forcing function to add real cards in the
next cycle.

### 3.5 Sequence number strategy

- Each side maintains an outbound `seq` counter per **logical
  session** (not per WS connection); reconnect does not reset.
- **Server-side replay buffer (G2c: F-04).** Per-session in-memory
  ring buffer of the last **500 `chat.event` frames** (also 5 MB
  serialized cap; whichever trips first evicts oldest). Each
  entry `{seq, frame}` is appended synchronously *before*
  `socket.send`, so survives WS death mid-send. Lives on the
  session, not the connection. The JSONL fsync path is for
  *terminated* invocations; this buffer is for *mid-stream*
  reconnects — independent.
- On reconnect, client sends `session.request_state{lastSeenServerSeq}`.
  Server responds:
  - `lastSeenServerSeq === serverSeq` → `session.state{gapDetected:false}`,
    resume live.
  - gap within buffer → replay buffered frames in order, then
    `session.state{gapDetected:false}`, resume live. Replayed
    frames are idempotent by seq.
  - gap exceeds buffer → `session.state{gapDetected:true}`;
    client re-fetches via `history.get?replay=true` (durable
    JSONL path).
- Server outbound `seq` is monotone for the lifetime of the
  process; SQLite persists `session.last_server_seq` so a server
  restart preserves ordering. After a server restart the in-memory
  buffer is empty, so any active session takes the
  `gapDetected:true` JSONL branch on first reconnect.

Server-initiated close codes:

| Code | Reason | Client retries? |
|---|---|---|
| 1000 | Normal closure (server shutdown explicit no-restart) | no |
| 1001 | Going away (process exit) | yes |
| 1008 | Policy violation (e.g. zod rejected the frame's basic shape) | no — terminal |
| 1011 | Internal error (uncaught server) | yes |
| 1012 | Service restart (SIGTERM/SIGINT drain) `[brief § 7]` | yes |
| 4000 | Frame failed Zod validation (semantic, not basic) | no — terminal |
| 4001 | Sequence-number replay rejected | yes |
| 4002 | Session expired / superseded | no |

The 4000/4001/4002 range matches the resilient-ws-ui retriable-vs-permanent
split `[ws R7]`: 4000 and 4002 are permanent (don't loop on a logic bug);
4001 is retriable (client just re-syncs).

## 4. Persistence schema

**Decision: in-DB rows for `session`/`invocation` summaries + JSONL on
disk for the full event stream per invocation.**

Rationale:
- The History tab list view is summary-only and uses SQL filter/sort
  over hundreds–thousands of rows. SQLite indexes are the right
  structure.
- The detail view re-renders a transcript whose size is unbounded
  (long sessions emit thousands of events). Storing event blobs
  inside SQLite would inflate the DB unboundedly and slow `VACUUM`.
- JSONL on disk supports append-only writes during a live session
  with one `fs.write` per event — exactly what we need for the
  "live updates while chat runs" wire frame `[brief § 6]`.
- Drill-down detail view streams JSONL back over WebSocket via the
  `history.replay_event` frames. This is the same renderer as
  Chat, so format parity is automatic.

Schema (DDL):

```sql

PRAGMA foreign_keys = ON;
PRAGMA synchronous = NORMAL;
PRAGMA busy_timeout = 5000;

CREATE TABLE IF NOT EXISTS schema_version (

);

  id                    TEXT    PRIMARY KEY,           -- UUID v4

  sdk_session_id        TEXT                            -- the SDK's own UUID, for resume
);
CREATE INDEX IF NOT EXISTS idx_session_started_at ON session(started_at DESC);
```

Schema migrations: `schema_version` table + `migrations.ts` array of
`{ version: 1, up: string }`. M5 contains one row; future versions
append. Down-migrations are out of scope; we trust forward-only.

JSONL event-log format:
- One file per `invocation.id` at `./var/db/events/<invocationId>.jsonl`.
- Each line is one `SDKMessage` JSON object, in receipt order.
- Lines are appended synchronously inside the SDK→WS bridge so a
  crash cannot lose more than the in-flight line.
- File is closed (fsync) when the invocation reaches a terminal state.

## 5. SDK integration spike findings

Read into `/tmp/exchange/cq-spike/node_modules/@anthropic-ai/claude-agent-sdk/`:

### 5.1 Entry point

`query({ prompt, options })` returns a `Query` interface which extends
`AsyncGenerator<SDKMessage, void>` `[sdk:sdk.d.ts:Query @ 2118]`. The
async iteration is how we receive the live stream. The server bridges
this generator to the WebSocket: for each yielded `SDKMessage`, emit
a `chat.event` frame after Zod-validating the envelope.

### 5.2 Sub-agent invocations — how they surface (resolves brief § 5 question)

The SDK exposes **two complementary signals** for sub-agent
invocations:

1. **Message-level annotation.** Every `SDKAssistantMessage` and
   `SDKUserMessage` carries `parent_tool_use_id: string | null` plus
   optional `subagent_type` and `task_description`
   `[sdk:sdk.d.ts:SDKAssistantMessage @ 2556]`,
   `[sdk:sdk.d.ts:SDKUserMessage @ 3672]`. A `null` parent_tool_use_id
   = top-level "main" invocation; non-null = sub-agent message under
   the named tool_use.
2. **System-level lifecycle events.** The SDK emits three
   `SDKTask*Message` system messages
   `[sdk:sdk.d.ts:SDKTaskStartedMessage @ 3610]`,
   `[sdk:sdk.d.ts:SDKTaskProgressMessage @ 3588]`,
   `[sdk:sdk.d.ts:SDKTaskNotificationMessage @ 3570]`. Each carries
   `task_id`, optional `tool_use_id`, `subagent_type`, `description`,
   running `usage` totals, and `status: 'completed'|'failed'|'stopped'`
   at the notification step.

**Plus**, the SDK option `forwardSubagentText: true`
`[sdk:sdk.d.ts:Options @ 1485-1490]` opts into the *full* sub-agent
text + thinking blocks being forwarded as assistant/user messages
with `parent_tool_use_id` populated. **The brief requires this**
because of the "full live collapsible feed of [sub-agent's] own
stream" `[brief § 4]` and "transcript is fully captured (this is
what powers the History tab)" `[brief § 4]`. We will set
`forwardSubagentText: true`. Cost: more bandwidth/storage. Worth it.

Hooks `SubagentStart` / `SubagentStop` exist
`[sdk:sdk.d.ts @ 5593-5624]` and provide `agent_id`, `agent_type`,
`agent_transcript_path` (a JSONL the SDK already wrote!), and
`last_assistant_message`. These are **belt-and-suspenders**: we wire
them as a fallback for invocation-row insertion so we never miss a
sub-agent, even if `forwardSubagentText` ever drops events. Per
`[brief § 5]` "Hooks are an acceptable fallback."

Also `agentProgressSummaries: true`
`[sdk:sdk.d.ts:Options @ 1641-1650]` — periodic ~30s summaries on
`task_progress` events. We enable this for live-row updates in the
sidebar pinned-list `[brief § 4 TaskCreate/TaskList/TaskUpdate]`.

**Conclusion:** sub-agent events are first-class in the SDK; no
escalation is needed. This was the brief's named risk
`[brief § 10 algedonic carve-out 3]` and it is resolved.

### 5.3 Abort/cancel semantics

`Query.interrupt(): Promise<void>` `[sdk:sdk.d.ts:Query @ 2128]` is
the documented control-channel method. Only valid in **streaming
input mode** (i.e. when `prompt` is an `AsyncIterable<SDKUserMessage>`,
not a string).

**Decision:** we use streaming-input mode unconditionally. The user
input arrives via WebSocket; we feed the SDK with an
`AsyncIterable<SDKUserMessage>` whose iterator is driven by the
queue of incoming `chat.input` frames. This is the only way to
support `chat.interrupt` per the brief.

In addition, the SDK exposes:
- `Query.stopTask(taskId)` `[sdk:sdk.d.ts:Query @ 2322]` — stop a
  specific sub-agent / Bash task.
- `Query.backgroundTasks(toolUseId?)` `[sdk:sdk.d.ts:Query @ 2335]` —
  background a foreground task (Ctrl+B semantics). Not exposed in
  v1 UI but kept available for M5 polish.
- `Query.close()` — hard-terminate, used in graceful shutdown.

For abort safety, every `query()` call also receives an
`AbortController` in `Options.abortController` so a SIGTERM drain
can both `abort()` and `interrupt()` for belt-and-suspenders.

### 5.4 Hook surface

`HOOK_EVENTS` `[sdk:sdk.d.ts:HOOK_EVENTS @ 780]` is a 29-event union.
We use **only** `SubagentStart` and `SubagentStop` in v1 (per
`[brief § 3]` plugins/hooks out of scope **except** as fallback per
`[brief § 5]`). All other hook events ride through `chat.event` if
`includeHookEvents: true` — we leave that false in v1 to keep the
stream small.

### 5.5 MCP forwarding

`Options.mcpServers: Record<string, McpServerConfig>`
`[sdk:sdk.d.ts:Options @ 1547-1560]`. The user's `~/.claude/`
configuration is already understood by the SDK at the CLI/subprocess
level — when we run the SDK via its bundled executable
(`pathToClaudeCodeExecutable` defaults to the SDK's bundled binary),
the subprocess inherits user-level settings from the same files
Claude CLI reads. So **we do not have to re-read `~/.claude/*.json`
ourselves**; the bundled CLI does. We only need to pass any
explicit per-session overrides via `mcpServers`. For v1, no overrides
— the user's existing config is forwarded automatically.

`Query.mcpServerStatus(): Promise<McpServerStatus[]>`
`[sdk:sdk.d.ts:Query @ 2211]` returns the live status; we surface
this in the chat header tooltip.

### 5.6 Skills + slash commands enumeration

`SDKSystemMessage` (subtype `'init'`) carries `skills: string[]`,
`slash_commands: string[]`, `mcp_servers`, `tools`, `model`, `cwd`,
`permission_mode` `[sdk:sdk.d.ts:SDKSystemMessage @ 3534-3568]`. This
fires once at session start.

`Query.supportedCommands(): Promise<SlashCommand[]>`
`[sdk:sdk.d.ts:Query @ 2193]` and `Query.supportedAgents()`
`[sdk:sdk.d.ts:Query @ 2205]` are also available mid-session. The
slash-command autocomplete uses `init.slash_commands` as the source
and re-fetches via `supportedCommands()` on `ConfigChange` hook
(deferred to M5 — for M3 we lock at init).

### 5.7 Permission prompts

`Options.canUseTool` (`sdk.d.ts:188`) fires before tool calls when
permission rules don't pre-decide. Context includes `title`,
`displayName`, `description`, `toolUseID`, `agentID`; returns a
`PermissionResult` (`sdk.d.ts:1953`) with
`behavior:'allow'|'deny'`. Async — bridge mints
`permissionRequestId`, emits `chat.permission_request`, parks a
Promise, resolves on `chat.permission_reply`. AbortSignal-aware:
WS disconnect rejects the Promise.

Wire mapping: `allow_once → {behavior:'allow'}`; `allow →
{behavior:'allow', updatedPermissions: suggestions}`; `deny →
{behavior:'deny', message:'User denied'}`.

### 5.7.1 Elicitation (G2c: F-01)

`Options.onElicitation: OnElicitation` (`sdk.d.ts:1208`,
`:1430`) fires when an MCP server requests structured user input.
Control_request subtype `elicitation` (verified in
`sdk.mjs:processControlRequest`). `ElicitationRequest.mode:
'form' | 'url'` (`sdk.d.ts:554`):

- **`form`** — `requestedSchema` describes fields; callback returns
  `{action:'accept', content:{...}} | {action:'decline'} | {action:'cancel'}`.
- **`url`** — `url` opens in the browser for an OAuth-style flow;
  callback waits for `SDKElicitationCompleteMessage`
  (`sdk.d.ts:388`) and returns `{action:'accept'}`.

Implementation mirrors `canUseTool` (PR-28): bridge mints
`elicitationId`, emits `chat.elicitation_request`, parks Promise,
resolves on `chat.elicitation_reply`. URL mode: client opens
`window.open(url, '_blank', 'noopener')`, shows "Waiting…"; bridge
ties Promise to `SDKElicitationCompleteMessage` by elicitationId.
A hook-registered `ElicitationResult` handler runs first if present
(`sdk.d.ts:1415`).

### 5.7.2 AskUserQuestion (G2c: F-02 re-spike outcome)

**Status: ESCALATE-TO-USER** (see § 11 Q-1). Re-spike findings:

1. AskUserQuestion is a built-in SDK tool with structured
   input/output (`sdk-tools.d.ts:608`, `:2760`).
2. The only Option is `toolConfig.askUserQuestion.previewFormat`
   (`sdk.d.ts:5729`); `sdk.mjs` forwards it as the env var

3. `Query` (`sdk.d.ts:2118-2345`) exposes no `respond` / `answer`

4. The six control_request subtypes the SDK handles (`sdk.mjs`

   `host_auth_token_refresh`, `hook_callback`, `mcp_message` —

5. The CLI subprocess runs with `--input-format stream-json
   --output-format stream-json` (`sdk.mjs initialize()`); stdin

**Candidates:**

- **C (escape hatch).** Pass `disallowedTools:

PR-31 spikes Candidate A at build time. If it fails, § 11 Q-1

## 7. Cross-cutting concerns

### 7.1 Graceful shutdown

- **Server SIGTERM/SIGINT:** drain in-flight SDK query (interrupt +
  abort) + close WS sockets with 1012 + close DB. PR-48 (G2c: F-06
  renumbered).
- **Client BFCache:** `pagehide(persisted)` closes sockets; `pageshow
  (persisted)` reopens. PR-10.

### 7.2 Logging schema

- Format: JSON lines (one object per line) `[brief § 7]`.
- Required fields: `ts` (ms epoch), `level` (debug|info|warn|error),
  `msg`, `comp` (component name).
- Optional fields per call site.
- File: `./var/log/cq-YYYYMMDD.log`, daily rotation on first write
  of new day (no compaction in v1).
- Stdout mirror at the same level.
- Level defaulted to `info`; `--log-level debug` flag toggles.

### 7.3 Accessibility commitments

- Every interactive element has either visible text or an
  `aria-label`.
- Focus visible at all times (no `outline: none` without a substitute).
- `prefers-reduced-motion: reduce` disables countdown-ring animation
  and toast slide-in.
- Connection indicator uses three channels: color + ring shape +
  text label (in tooltip and aria-label). `[ws V3]`.
- Streaming text uses `aria-live="polite"` so screen readers
  announce updates without barging in.

### 7.4 Error UX

- Bounded toast log (50 entries; render top 5 visible, rest accessible
  via "show all errors" expansion). PR-49 (G2c: F-06 renumbered).
- Every `chat.error` frame produces a toast.
- Every WS close-code in the 4xxx range produces a toast labelled
  with the close code text.
- Connection indicator surfaces transport failures `[ws V10]`.

### 7.5 Session-id allocation

- Server allocates `session.id` and the top-level `invocation.id`
  on `chat.start`, using `crypto.randomUUID()` (Bun-native). Sub-agent
  invocations get `invocation.id` allocated by the server on
  `task_started` reception, NOT by the SDK's `task_id` (which we
  store separately as a denormalized field).

### 7.6 Time / clock sources

- Server uses `Date.now()` for ledger timestamps; persisted as
  integer ms epoch.
- Wall clock skew between server and client is irrelevant — heartbeat
  RTT is measured via the server *echoing* `clientTs`
  `[ws R3]`, so client computes its own RTT without relying on
  server time.
- Time-jump detector uses `Date.now()` differences only `[ws R8]`.

### 7.7 CLI flags

```
cq [options]
  --cwd <path>            Working directory exposed to the agent.
                          Default: process.cwd()
  --host <addr>           Bind address. Default: 127.0.0.1
                          (per [brief § 3] "VPN interface" → user sets explicitly)
  --port <n>              HTTP+WS port. Default: 5173
  --db <path>             SQLite path. Default: ./var/db/cq.sqlite
  --log-level <level>     debug|info|warn|error. Default: info
  --shutdown-timeout-ms <n> Default: 5000
  --model <id>            Default model for new sessions.
                          Default: SDK's default
  --help                  Print this help.
```

All flag parsing in `main.ts` is hand-rolled (~30 lines). No
argparser dependency `[ccmd §4]`.

## 10. Risks & open assumptions

### Risks deliberately accepted

1. **Bun version drift.** Bun moves fast. Pin to 1.3.13 in the flake;
   re-evaluate at M5 if a 1.4.x ships with a relevant fix. Mitigation:
   pin via flake, not just `package.json#engines`.
2. **SDK version drift.** `@anthropic-ai/claude-agent-sdk` is at
   0.x and major shape changes between minor versions are plausible.
   Pin to 0.3.150 and re-validate on every patch bump. Mitigation:
   ship a "smoke test" CI job that verifies the SDK still emits
   `subagent_type` and `parent_tool_use_id` — if it stops, fail
   loud.
3. **Sub-agent transcript volume.** `forwardSubagentText: true` can
   inflate event volume 5-10× on deep recursions. Mitigation: per-frame
   bandwidth budget on the WS is bounded by Bun's WebSocket send
   buffer; the bridge applies backpressure (await `socket.send`
   before continuing iteration).
4. **JSONL write fsync cost.** Synchronous append per event will
   limit throughput on slow disks. Mitigation: buffer with a small
   `pending: SDKMessage[]` and flush every 16ms or 4KB; fsync only
   on invocation close.
5. **Bun:sqlite vs. better-sqlite3 surface differences.** We use
   `bun:sqlite` only; no Node fallback. Acceptable per `[brief § 3]`.
6. **Permission prompt latency.** If the user takes minutes to
   respond to a permission prompt, the SDK is parked. The brief
   allows this; no timeout — surfacing a "waiting for response"
   indicator in the card is sufficient.
7. **Cost computation is the SDK's, not ours.** We forward
   `result.usage` from the SDK; we do not re-derive. If the SDK's
   numbers are wrong, ours are wrong identically — that's preferable
   to drift between cq's number and CLI's.
8. **Background-tab heartbeat throttling (G2c: F-09 / `[ws R14]`).**
   Chrome throttles timers to 1/min after ~5min hidden + ~30s
   silent; heartbeats may stop firing in background tabs. We
   accept this: the time-jump detector (PR-11) recovers on
   visibility return. Web Worker heartbeat would prevent the
   gap; not v1. README surfaces this.

### Open assumptions

A1. The bundled CLI honours `~/.claude/` config (settings, MCP,
    skills) via `query()` with no per-call override. Inferred from
    `resolveSettings @ 2414` + absence of "disable" flag.
    **To-be-confirmed empirically at PR-19's
    `mcp-inheritance.test.ts` (G2c: F-14).** Fallback if it fails:
    PR-19 reads `~/.claude/mcp_servers.json` explicitly and passes
    via `Options.mcpServers`; A1 dropped.

A2. `forwardSubagentText: true` + `agentProgressSummaries: true`
    is the right combination to render nested SubagentCards with
    live progress. **Plan commits to this combination.** If field
    experience shows over-noisy stream, M5 can add a toggle.

A3. `Query.readFile(path, options)` (the file-reference expansion
    in PR-34) is gated by the SDK's read-permission rules. We do
    not need to re-implement permission gating for path:line
    anchors — trust the SDK.

A4. CSS-modules with Bun's bundler works without per-file extra
    config. **To validate at PR-13.** If broken, fall back to
    `*.module.css` → plain CSS class names compiled by hand
    (still no extra dep).

A5. `bun test`'s happy-dom integration supports `requestAnimationFrame`
    (PR-14) and `document.title` (PR-16). **To validate at PR-14.**
    Both are widely-used DOM APIs and happy-dom is mature; if missing,
    we use `vi-style fake-rAF` shim (~10 lines).

A6. The brief's "two tabs in the UI, no more for v1" is exhaustive.
    Modals, popovers, pickers (slash autocomplete, resume picker,
    permission prompt, AskUserQuestion form) are not "tabs" per
    `[brief § 3]` — they are inline UI inside the Chat tab. **Plan
    commits to this reading.**

A7. The user's working directory `--cwd` is read-write for the
    agent (Read/Write/Edit tools). The brief does not say so
    explicitly, but a UI that locked the SDK out of writes would
    not "replicate CLI experience" `[brief § 4]`. **Plan commits
    to read-write.** The user controls risk by choosing the cwd.

A8. Permission-mode toggle mapping (G2c: F-03 corrected):
    yolo → `bypassPermissions` (with safety flag); standard →
    `default`; plan → `plan`; **read-only → SDK `default` + cq's
    `canUseTool` deny overlay (PR-29)**. The prior read-only →
    `dontAsk` mapping was wrong: `dontAsk` only suppresses prompts
    (`sdk.d.ts:1589, 1929`), pre-approved writes still execute.
    `auto` / `acceptEdits` live in an "advanced" sub-menu.

A9. WS upgrade only on path `/ws`; rest is static. **Origin check
    (G2c: F-19):** PR-06's handler rejects upgrades (1008) when
    Origin's host:port ≠ Host. Browsers always send Origin; defeats
    trivial cross-tab drive-bys. No CORS, no token auth (brief:
    VPN binding). **Plan commits.**

### Places downstream cycles may need to escalate

- If `forwardSubagentText` ever drops events (regression in SDK),
  the SubagentStart/SubagentStop hook fallback must be activated
  → S4 fires for that cycle.
- If a future SDK release adds new `SDKMessage` variants between
  M2 and M5, the passthrough schema absorbs them with "unknown SDK
  event vN" cards (§ 3.4) but a new dedicated card may be desired
  → S4 / planner refresh.
- If Bun-native CSS-modules turn out broken (A4), fall back to
  hand-rolled CSS class names. No escalation needed.
- If JSONL volume saturates disk on a long test (M4), partition
  events by hour or by 100MB chunks. No escalation needed; planner
  refresh inside M4.
