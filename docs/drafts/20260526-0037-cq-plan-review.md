# G2b adversarial review — cq plan

Plan reviewed: `docs/drafts/20260526-0037-cq-plan.md` (1998 lines)
Reviewer: G2b S3* adversarial pass

## Verdict

**ACCEPT-WITH-FIXES.**

The plan is comprehensive, well-structured, and demonstrably grounded
in an SDK spike (load-bearing claims spot-verified, see Audit-spot-checks
below). § 1 stack pinning, § 3 wire schemas, § 4 persistence DDL, and
§ 6 PR breakdown all carry enough detail that an executor can act on
them. Resilient-WS-UI Part-3 reliability bullets are densely mapped to
PRs PR-06 … PR-18 with named test files and concrete assertions.

However, several MAJOR gaps must be patched before the planner hands
off to M2-onwards executors: (1) MCP **elicitation** is declared in
the wire protocol and the file layout but has no PR that delivers
either the server-side `onElicitation` handler or the client UI;
(2) the AskUserQuestion answer path is described as "fabricated
tool_result … same mechanism as canUseTool reply" — these are
different SDK surfaces and the plan's spike never names the actual
mechanism (PR-29 has unsubstantiated acceptance criteria); (3) the
permission-mode mapping `read-only → dontAsk` (A8) is wrong against
the SDK's own definition; (4) mid-stream reconnect / event replay
(brief § 6 sequence-number requirement) is under-spec'd — `gapDetected`
flips a boolean but the plan never says how the *current invocation's*
already-emitted events get retransmitted while the JSONL writer is
still buffering for fsync (risk 4); (5) graceful-shutdown's 5s drain
contract isn't reflected in PR-46's acceptance test. These are not
re-plan-the-whole-thing problems but each blocks a specific PR's
acceptance.

## Severity scale

- **MAJOR** — plan ships defective work unless fixed before code starts.
- **MINOR** — plan ships work that will need follow-up but won't break a milestone.
- **NIT** — cosmetic / preference; surface but do not block.

## Findings

### F-01 — Elicitation has schemas + file but no PR — [MAJOR]

- **Location**: plan § 2 (`agent/elicitation.ts` listed); § 3.3 `ChatElicitationReply`, § 3.4 `ChatElicitationRequest`; § 6 M3 PR-27…PR-37 (none deliver elicitation); § 8 test list mentions "permission/elicitation roundtrips".
- **Observation**: The wire protocol declares two elicitation frames and the file layout lists `packages/server/src/agent/elicitation.ts`, but **no PR in any milestone implements them**. PR-28 covers `canUseTool` only. There is no UI component for elicitation in the file layout under `packages/web/src/chat/`. There is no test file in § 8 named for elicitation despite the prose mentioning it.
- **Evidence**: SDK `sdk.d.ts:1208` (`OnElicitation = (request, options) => Promise<ElicitationResult>`); `Options.onElicitation` is a separate callback from `canUseTool` (`sdk.d.ts:188`). They cannot share implementation. SDK also exposes `ElicitationRequest.mode = 'form' | 'url'` (`sdk.d.ts:554`) — URL mode requires opening a browser window for OAuth-style flows, which the plan's "MCP forwarding" claim (§ 5.5) does not address.
- **Impact**: M3 is "Chat full fidelity" per brief; without elicitation, an MCP server that asks for input (e.g. an OAuth flow, structured form) stalls the SDK indefinitely or produces an unhandled `chat.event` with no UI. PR-26's M2 acceptance still passes (no MCP triggers elicitation in stub), but real-world use breaks.
- **Suggested fix**: Add PR-28b (or split PR-28) **"Elicitation roundtrip: `onElicitation` callback + `ChatElicitationCard.tsx` + URL-mode handler"** with acceptance: feed a synthetic ElicitationRequest (form mode) through bridge, assert WS frame, reply, assert Promise resolves with `ElicitResult`. Then a second case for URL-mode that opens a new tab to the URL and waits for completion via `SDKElicitationCompleteMessage` (`sdk.d.ts:388`). Add `Cards/ElicitationCard.tsx` to the M3 file layout.

### F-02 — AskUserQuestion answer path is hand-waved — [MAJOR]

- **Location**: plan § 6 PR-29 "Server maps `chat.question_reply` to a fabricated tool_result for the corresponding tool_use_id (the SDK consumes it via the same mechanism as canUseTool reply)".
- **Observation**: `canUseTool` is an **authorization** callback that returns a `PermissionResult` *before* a tool runs (`sdk.d.ts:188`, `:1953`). It does not inject tool_results into the message stream. AskUserQuestion is a built-in tool the SDK executes inside the bundled CLI subprocess; the actual answer mechanism in the SDK is not the `canUseTool` callback. The plan's spike (§ 5) did not establish *how* an SDK consumer feeds an AskUserQuestion answer back — only that `toolConfig.askUserQuestion.previewFormat` exists (`sdk.d.ts:5729`).
- **Evidence**: Re-read SDK shows AskUserQuestion is in the SDK's own tool set (executed inside the CLI subprocess); the SDK consumer's only injection point into the model conversation is the streaming-input `AsyncIterable<SDKUserMessage>`. The plan's claim "the SDK consumes it via the same mechanism as canUseTool reply" is unsupported by any cited line; § 5.7 is about permission, not AskUserQuestion.
- **Impact**: PR-29 will land code that doesn't connect: the bridge sends a fabricated tool_result, but the SDK is not listening for one on that channel. The acceptance test as written ("server emits the synthetic tool_result") will pass against a stub but fail in real SDK integration. M3 ships broken UI for one of the brief's named features.
- **Suggested fix**: Spike the actual AskUserQuestion answer path before scheduling PR-29 — likely candidates: (a) push a synthetic `SDKUserMessage` of role 'user' with a `tool_result` content block via the streaming-input queue; or (b) a dedicated control method on `Query` that the spike missed (search for `respond`, `answer`, `submitToolResult` in `sdk.d.ts`). Rewrite PR-29 with the verified mechanism cited (file:line). If (a) is correct, document the message shape; if neither, escalate per § 10 carve-out 3 ("a finding that the Agent SDK lacks a capability needed for an acceptance item").

### F-03 — Permission-mode mapping `read-only → dontAsk` is wrong — [MAJOR]

- **Location**: plan § 10 A8.
- **Observation**: Plan maps brief's "read-only" to SDK's `dontAsk`. The SDK defines `dontAsk` as "Don't prompt for permissions, deny if not pre-approved" (`sdk.d.ts:1589, :1929`). That is *not* read-only; a session in `dontAsk` mode *can* write if write permissions were pre-approved. A true read-only mode would deny *all* write/edit/bash and allow read/grep/glob — there is no SDK mode that does this out of the box.
- **Evidence**: `sdk.d.ts:1931`: `PermissionMode = 'default' | 'acceptEdits' | 'bypassPermissions' | 'plan' | 'dontAsk' | 'auto'`. None of these is named "read-only". The closest *semantic* match for "read-only" is `'plan'` ("Planning mode, no actual tool execution"; `sdk.d.ts:1588`) — but `plan` also blocks reads.
- **Impact**: When the UI toggle says "read-only", the user expects safety from any modification. Setting `dontAsk` provides the opposite: if any pre-approved rules exist (and the brief's MCP forwarding will pull user-level rules from `~/.claude/`), edits succeed silently. This is a safety regression vs CLI.
- **Suggested fix**: Either (a) drop "read-only" from the toggle and expose the four real SDK modes (`default`, `acceptEdits`, `bypassPermissions`, `plan`); (b) implement "read-only" as a cq-level overlay: keep SDK in `default` mode, intercept `canUseTool` for any write/edit/bash tool, return `behavior: 'deny'`; (c) escalate to the user that brief § 4's "read-only" cannot be honored without (b). Recommend (b) and add a PR to M3 (split out of PR-28) to implement the overlay. Update A8 with the new mapping.

### F-04 — Mid-stream reconnect / sequence-gap recovery is under-spec'd — [MAJOR]

- **Location**: plan § 3.5; § 4 ("JSONL on disk"); § 10 risk 4 ("buffer with a small pending: SDKMessage[] and flush every 16ms or 4KB").
- **Observation**: Brief § 6 mandates "sequence numbers per session so a reconnect can resume mid-stream without dropping events". Plan's mechanism: client sends `session.request_state` with `lastSeenServerSeq`; server returns `gapDetected: true` if a gap exists; client must re-fetch via `history.get`. But `history.get` reads JSONL from disk, and § 10 risk 4 explicitly says JSONL writes are *batched* every 16ms/4KB. Therefore the events that the client missed (between disconnect and reconnect) may not yet be on disk, especially for the live invocation. There is no buffer of unacked frames on the server side. Result: mid-stream reconnect drops events.
- **Evidence**: § 3.5 says "the client re-fetches missing transcript via `history.get` with `replay: true`"; § 4 says JSONL file is "closed (fsync) when the invocation reaches a terminal state"; § 10 risk 4 adds the batching. No PR specifies how the bridge ensures every emitted `chat.event` is durable before its seq counter advances.
- **Impact**: Brief § 6 acceptance ("a reconnect can resume mid-stream without dropping events") is not met. Brief § 11 stop-condition requires the stream → tool cards → invocation in History to round-trip; with batched writes, an interrupted reconnect can lose tail events.
- **Suggested fix**: Choose one: (a) hold a ring buffer of last-N `chat.event` frames per session in memory (N sized to expected gap; ~500 frames), index by `seq`, replay on `session.request_state` matching the gap; or (b) make JSONL writes synchronous (`O_APPEND` + per-line `fs.fsync`) and walk back the file by ordinal in `history.replay_event` even mid-stream. Add a PR in M1 (after PR-09) called **"server-side seq replay buffer"** with acceptance: client disconnects after `seq=5`, server emits frames 6-15 to the void, client reconnects, server replays 6-15 on `session.request_state`.

### F-05 — Graceful shutdown timeout is unverified — [MINOR]

- **Location**: plan § 6 PR-46; § 7.1; § 10 risk M5 (2).
- **Observation**: PR-46 says "wait up to 5s for `chat.done`" and acceptance is "observes a clean exit within 6s." But there is no acceptance criterion for the 5s drain — what happens if `chat.done` never arrives? The plan exposes `--shutdown-timeout-ms` (§ 7.7) but PR-46 doesn't test the timeout-exceeded path (forced abort + 1012 anyway).
- **Evidence**: Brief § 7 "Graceful shutdown: SIGTERM/SIGINT drain in-flight SDK queries (abort + flush) and close WebSocket sockets with a `1012 Service Restart` code". The plan's PR-46 covers the happy path; the worst-case (SDK hangs past timeout) is mentioned in M5 risks but not tested.
- **Impact**: A real SIGTERM in production may hang the process past systemd's default 90s `TimeoutStopSec`, leading to SIGKILL and a hard close (1006, not 1012) — client mistakes the cause.
- **Suggested fix**: Add a second acceptance case to PR-46: "shutdown when SDK does NOT respond to interrupt within timeout still closes WS with 1012 and exits within shutdown-timeout-ms + 500ms." Drive the test by setting `--shutdown-timeout-ms 200` and using a deliberately-unresponsive SDK stub.

### F-08 — PR-02 acceptance "≥ 25 tests; one per frame type minimum" is a count, not a property — [MINOR]

- **Location**: plan § 6 PR-02.
- **Observation**: Acceptance criterion is a test count and "round-trip parse … AND a rejection for one targeted invalid field per schema". For 25+ frames, "one targeted invalid field per schema" can hide structural gaps — e.g. the test for `ChatInput` could reject an invalid `mimeType` but never test that `attachments` total over 5MB is rejected. The 5MB cap is asserted in § 3.3 but is not a schema-level constraint (it's a size-of-base64 check), so it must be tested separately.
- **Evidence**: § 3.3 "Attachments are capped at 5 MB total per `chat.input` frame, enforced both client- and server-side"; PR-02 acceptance doesn't name this assertion.
- **Impact**: PR-02 passes with the 5MB cap unimplemented; the cap shows up in PR-33 (attachments) but not on the server. A 50MB paste gets to the server before being rejected.
- **Suggested fix**: Augment PR-02 acceptance with: "test asserts that a `ChatInput` with `attachments[].dataBase64` summing to >5MB is rejected by the validator (either Zod refinement or a separate `validateAttachmentSize(frame)` helper that PR-02 ships)." Cite the test name `protocol.test.ts: ChatInput rejects oversize attachments`.

### F-09 — Resilient-WS-UI R14 (Web Worker for heartbeat) is not addressed — [MINOR]

- **Location**: plan § 6 PR-08; § 10 risks.
- **Observation**: Resilient-WS-UI § R14 calls out: "Chrome throttles main-thread timers to 1/min after 5 min hidden + 30s silent. … if you need heartbeats to keep running while hidden, use a **dedicated Web Worker** for the heartbeat timer." The plan defers-while-hidden (PR-10, Phoenix pattern) which is fine for app-level reconnect, but does NOT address the case where the user keeps cq open in a background tab during a long-running SDK query and relies on cq's heartbeat to detect disconnects. With timer throttling, the client's nonce timeout won't fire promptly, and the time-jump detector (PR-11) will fire on resume — that's a recovery, not a prevention.
- **Evidence**: Skill § R14 is in Part 1 (reliability), not Part 3 checklist; plan addresses every Part 3 item but never explicitly decides whether to ship a Web Worker.
- **Impact**: Long-running queries in background tabs may show stale state until user returns. Brief § 4 "Live token + cost counters" updates stall in this case.
- **Suggested fix**: Add a one-line "we accept R14 as a known limitation; time-jump detector is the recovery path" to § 7 (Cross-cutting concerns), citing R14. Or add a small PR in M5 to host heartbeat in a dedicated Worker, with acceptance: timer continues firing at 15s interval after 6min in hidden tab (test uses `chrome.test.simulateBackgroundThrottle` or equivalent).

### F-12 — `bun build --watch` ≠ HMR — [MINOR]

- **Location**: plan § 0 ("Both serve from the same `Bun.serve`"); § 6 PR-05 ("`dev` script using `bun build --watch` for the web package").
- **Observation**: `bun build --watch` (verified `bun build --help`) is rebuild-on-file-change with full page reload (or process restart); it is not Hot Module Replacement. Plan calls this "HMR for the React app" (§ 0 of brief, § 0 of plan). Bun 1.3 does support frontend HMR via `Bun.serve`'s integrated dev mode (`bun --hot`), but the plan picks the wrong incantation.
- **Evidence**: `bun build --watch` rebuilds; `bun --hot` enables HMR for server processes; frontend HMR requires `Bun.serve` integrated dev mode (Bun 1.2+ feature).
- **Impact**: Developer experience degrades; PR-05 acceptance ("Killing dev with SIGINT exits 0") passes regardless of whether HMR works. Naming creates an unmet expectation.
- **Suggested fix**: Either (a) drop the HMR claim, call it "live reload"; or (b) use Bun's integrated dev/HMR by serving the entrypoint via `Bun.serve({ development: { hmr: true } })` (Bun 1.3 API). Add a PR-05 acceptance: "edit `App.tsx`, observe the bundle re-emits within 500ms and the open page applies the change (with or without state preservation; document which)."

### F-13 — PR-39's "100 events per invocation … FTS search by prompt excerpt" doesn't test the trigger plumbing — [MINOR]

- **Location**: plan § 6 PR-39; § 4 ("Triggers to keep FTS in sync (omitted here; in code).").
- **Observation**: FTS5 in `external content` mode (`content='invocation'`) requires explicit triggers (`AFTER INSERT/UPDATE/DELETE`) to populate the FTS index; the plan defers them to "in code" without naming a test. PR-39 acceptance includes "FTS search by prompt excerpt" but doesn't say whether the test exercises an *update* path that would require trigger correctness.
- **Evidence**: SQLite FTS5 docs require trigger maintenance for external-content tables; otherwise FTS is permanently stale after the first row.
- **Impact**: PR-39 ships with FTS that works for INSERT but breaks on UPDATE (e.g. updated `prompt_excerpt` when title changes); History tab search returns stale matches.
- **Suggested fix**: Augment PR-39 acceptance with: "test inserts an invocation, updates its `prompt_excerpt`, asserts FTS search finds the *new* text and not the old." Name the test `persist-crud.test.ts: FTS updates reflect prompt_excerpt edits`.

### F-14 — § 5.5 claims user `~/.claude/` config is read by the bundled CLI subprocess without independent validation — [MINOR]

- **Location**: plan § 5.5; § 10 A1.
- **Observation**: Plan asserts: "when we run the SDK via its bundled executable (`pathToClaudeCodeExecutable` defaults to the SDK's bundled binary), the subprocess inherits user-level settings from the same files Claude CLI reads. ### F-15 — Test strategy doesn't apply [[dual-tests]] discipline beyond MockWebSocket / MockAnthropicHTTP — [NIT]

- **Location**: plan § 8 "Dual-tests pattern".
- **Observation**: Plan invokes [[dual-tests]] for the WebSocket and the Anthropic HTTP, which is the right call. But the *persistence* layer is tested only against real `bun:sqlite` (PR-39); per [[dual-tests]], the recommended pattern is an interface + hand-written in-memory dummy alongside the real adapter, so the dummy can drive `bridge.test.ts` (PR-19/20) at lower cost. Plan ships `bridge.test.ts` and then a *separate* `bridge-persist.test.ts` (PR-40) which is reasonable, but doesn't say the bridge has a `Persistence` interface that swaps between real-DB and in-memory implementations. Without that, the bridge code couples to `bun:sqlite` types.
- **Evidence**: § 2 file layout has `persist/sessions.ts` etc. but no `persist/Persistence.ts` interface; § 6 PR-39 doesn't introduce one.
- **Impact**: Bridge tests will pull `bun:sqlite` into every test that touches the bridge; slower test runtime, harder to isolate. Not a blocking issue.
- **Suggested fix**: Add a one-line note in § 2 that `persist/` exports a `Persistence` interface (the type-only contract) and a `SqlitePersistence` + `InMemoryPersistence` (defer to PR-39 to introduce, used by PR-40's tests). Optional.

### F-19 — No mention of CSP / origin restriction for WS upgrade — [NIT]

- **Location**: plan § 10 A9.
- **Observation**: A9 says "No CORS / Origin check (brief: no auth; same-origin from the browser is implicit; user binds to VPN)." The brief is explicit ("no auth, no CORS-permissive flows") — the planner chose to also drop Origin checking, which is defensible for VPN deployment, but means *any page the user visits with JavaScript* (e.g. a phishing page they visit in another tab) can open a WS to cq if they can guess the URL. - **Impact**: Low. Plan reading is defensible; flagged for transparency.
- **Suggested fix**: One-line note in § 10 A9 that the server checks `Origin: <bind-host>` matches the request Host (deny mismatches with 1008). Costs nothing, raises the bar marginally.
